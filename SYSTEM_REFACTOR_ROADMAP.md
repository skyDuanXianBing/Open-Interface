# 系统重构路线图

## 文档目的

本文档给出 Open Interface 当前系统的分阶段重构路线图，目标是在不破坏现有闭环可用性的前提下，逐步提升感知质量、定位精度、验证可靠性和恢复能力。

路线图分为三层优先级：

- P0：必须优先做，否则系统成功率和可维护性很快撞墙
- P1：显著提升稳定性和工程质量
- P2：面向更高阶 Agent 能力与长期演进

## 规划原则

### 1. 保持闭环可运行

任何阶段都不能为了重构而打断当前可运行闭环。优先采用“插拔式增强”，而不是一次性推翻。

### 2. 优先增强上层，不优先重写执行层

当前 `app/interpreter.py` 是相对稳定层。短期重点不在执行器大改，而在：

- 更好的感知输入
- 更强的定位精修
- 更可靠的验证
- 更显式的恢复策略

### 3. Prompt 优化不能代替系统能力建设

当前系统已经把 prompt 能做的事情做到了一个阶段性上限。后续收益更大的方向是：

- 提高输入质量
- 提升结构化信息质量
- 提高验证与恢复可编程性

### 4. 所有阶段都要可测量

每一阶段必须定义指标和验收标准，避免“感觉变好了”。

## 目标状态

期望把当前系统从：

> 弱语义视觉块 Agent MVP

逐步提升到：

> 具备结构化感知、局部精定位、语义验证与程序化恢复的桌面 Agent Beta

## P0：先把最核心的上限问题解决

### P0 目标

在不推翻现有架构的情况下，明显提升成功率上限，优先补齐感知、定位、验证、恢复四个短板中的最短板。

### P0-1 感知层增强：从弱视觉块升级到“弱语义感知”

#### 目标

让模型拿到的输入不再只是红框候选区域，而是包含更多语义线索。

#### 建议改造

1. 在 `app/utils/screen.py` 增加 OCR 通道
   - 对全屏或重点区域做文字抽取
   - 输出文字块 bbox、文本内容、置信度

2. 扩展 `screen_state`
   - 不再只总结 anchor 数量和区域大小
   - 增加文本块摘要、疑似输入框、疑似按钮、疑似窗口标题区域等

3. 引入候选区域类型字段
   - `region_type: text | icon | input_like | button_like | unknown`

4. 分离“原始视觉候选”和“语义增强候选”
   - 便于后续回退与对比

#### 预期收益

- 模型能更稳定识别输入框、按钮、弹窗标题、菜单项
- 减少仅靠红框猜位置带来的误操作

#### 风险

- OCR 可能带来性能开销
- OCR 输出噪声会污染 prompt

#### 验收标准

- `screen_state` 中出现文本级结构
- 输入框/按钮类场景的成功率相较当前 MVP 有可测提升
- 模型不再仅依赖 `visual_anchors` 作判断

### P0-2 定位层增强：增加局部精定位

#### 目标

把当前“粗 anchor -> 全局映射”的定位方式，升级为“粗定位 -> 局部精修”。

#### 建议改造

1. 增加局部二次裁剪
   - 选中 anchor 后，对局部区域做二次分析

2. 增加热点 refinement
   - 对小按钮、图标、菜单项进行局部中心重估

3. 给 anchor 增加“点击热点”字段
   - 不是只保留框中心，而是保留建议点击点

4. 对高密度区域增加特殊策略
   - 工具栏
   - 菜单栏
   - 桌面图标密集区

#### 预期收益

- 小控件命中率提升
- 工具栏、图标、密集按钮误点率下降

#### 风险

- 如果二次精修不稳定，可能比粗点更差

#### 验收标准

- 在固定样本集上，小热点点击成功率显著高于当前版本
- 误点率下降

### P0-3 验证层增强：从图像差分升级到“视觉差分 + 语义验证”

#### 目标

让系统不仅知道“画面变了”，还知道“任务是否真的朝目标推进了”。

#### 建议改造

1. 在 `app/verifier.py` 增加 OCR 对比
   - 输入前后文字变化
   - 标题变化
   - 弹窗文案变化

2. 引入轻量语义验证规则
   - `expected_outcome` 与 OCR/结构化状态做匹配

3. 区分“变化成功”和“变化但偏离目标”
   - 例如误弹窗、误切窗口、误聚焦

4. 补充 `uncertain` 分支的后续处理策略
   - 不再只是简单重试

#### 预期收益

- 降低“动作成功执行但任务没有推进”的假阳性
- 降低因无变化阈值导致的误判

#### 风险

- 语义验证规则初期可能覆盖不全

#### 验收标准

- 文本输入、窗口切换、弹窗确认场景中，验证准确性明显提升
- `failed` / `uncertain` 的分类更符合真实结果

### P0-4 恢复层增强：从 prompt 恢复升级到程序化恢复

#### 目标

把当前“让模型自己换策略”的隐式恢复，升级成系统可控的恢复框架。

#### 建议改造

1. 在 `app/core.py` 中引入恢复策略状态机
   - 同动作失败
   - 同区域失败
   - 同目标失败
   - 验证不确定

2. 明确恢复动作模板
   - 重新观察
   - 切换候选 anchor
   - 改动作类型
   - 扩大观察区域
   - 回退一步

3. 给 `agent_memory` 增加更明确的失败标签
   - `focus_failed`
   - `no_ui_change`
   - `wrong_window`
   - `target_not_found`

4. 引入恢复次数和恢复路径日志

#### 预期收益

- 降低盲目重试
- 提高复杂 GUI 场景恢复成功率
- 提升问题定位可解释性

#### 风险

- 恢复状态机设计过重会拖慢迭代

#### 验收标准

- 连续失败场景中，恢复路径可观察、可解释、可复盘
- 连续验证失败不再只等于停止，而是先走结构化恢复策略

### P0 交付物

- 增强版 `screen_state`
- OCR/语义感知基础能力
- 局部精定位机制
- 语义验证初版
- 恢复状态机初版

### P0 验收指标

- 基准任务集总体成功率提升
- 小控件误点率下降
- 验证误判率下降
- 平均连续失败次数下降

## P1：把 MVP 打磨成稳定可维护系统

### P1 目标

在核心能力上限提升之后，补齐模型路由一致性、测试体系、执行安全护栏和可观测性。

### P1-1 统一模型适配层协议

#### 目标

降低 `gpt5`、`gpt4o`、`gpt4v`、`computer-use-preview` 多路径维护成本。

#### 建议改造

1. 统一输入 payload 组装器
2. 统一输出 JSON 规范化器
3. 统一模型能力声明
4. 统一错误类型和超时行为

#### 验收标准

- 模型切换不需要在多个分支复制修补相同逻辑
- 输入输出协议可单独测试

### P1-2 测试体系重整

#### 目标

把当前混杂的校验脚本整理成清晰的测试层级。

#### 建议改造

1. 区分三类测试
   - 纯逻辑单测
   - 受控集成测试
   - 人工 GUI smoke test

2. 整理历史红灯脚本
   - 已过时的改为归档
   - 现行回归用例保留并修签名

3. 建立最小可靠回归集
   - `screen`
   - `interpreter`
   - `verifier`
   - `agent_memory`
   - `session_store`

4. 给关键接口加假对象协议

#### 验收标准

- CI 至少能稳定跑逻辑层和协议层测试
- 测试失败能反映真实回归，而不是接口漂移

### P1-3 执行安全护栏增强

#### 目标

降低桌面自动化的副作用与风险。

#### 建议改造

1. 增加前台窗口/焦点检测
2. typing 前增加目标区域确认
3. 清理或替代 `command` warm-up
4. 对破坏性动作增加更强约束
5. 对登录页、支付页、系统设置页增加硬性停机规则

#### 验收标准

- 焦点误判减少
- 无关窗口误操作减少
- 高风险页面能够稳定止损

### P1-4 观测与复盘能力增强

#### 目标

让失败可定位、成功可量化。

#### 建议改造

1. 为每步记录更完整的感知摘要、定位依据、验证结果、恢复路径
2. 给时间线增加结构化调试字段
3. 输出任务级统计指标

#### 验收标准

- 一个失败任务能在时间线中快速定位失败原因
- 可以量化不同阶段的成功率和失败类型

### P1 交付物

- 统一模型协议层
- 分层测试体系
- 执行安全护栏
- 结构化可观测能力

## P2：进入更高阶 Agent 能力建设

### P2 目标

在系统稳定后，逐步从“单步 reactive agent”走向“具备任务结构和长期记忆的桌面 Agent”。

### P2-1 任务级抽象

#### 建议改造

1. 引入子目标概念
2. 引入任务阶段状态
3. 支持“观察 -> 计划 -> 执行 -> 验证 -> 恢复”的任务图谱

#### 收益

- 复杂任务更稳定
- 能减少模型每一步都从零猜策略

### P2-2 长期记忆与策略复用

#### 建议改造

1. 跨请求保存高价值成功路径
2. 记录应用级操作习惯
3. 支持按应用场景注入经验模板

#### 收益

- 同类任务越做越稳
- 对固定软件流程适应更快

### P2-3 应用/窗口级专用插件化

#### 建议改造

1. 为浏览器、文件管理器、表格软件、聊天工具设计特化策略
2. 插件只增强，不破坏通用 Agent 主链路

#### 收益

- 高频场景成功率显著提升

### P2-4 指标驱动优化平台

#### 建议改造

1. 建立任务样本库
2. 建立成功率与失败分类看板
3. 建立版本对比基线

#### 收益

- 从“主观优化”转向“指标驱动优化”

## 推荐实施顺序

建议顺序如下：

1. P0-1 感知增强
2. P0-2 局部精定位
3. P0-3 语义验证
4. P0-4 程序化恢复
5. P1-2 测试体系重整
6. P1-1 模型适配统一
7. P1-3 执行安全护栏
8. P1-4 观测与复盘
9. P2 全部能力

原因是：

- 感知质量决定定位和验证上限
- 验证与恢复是闭环可靠性的核心
- 测试体系应尽早跟上，否则后续重构风险会快速放大

## 不建议的重构方式

以下路线不建议优先采用：

### 1. 先大改执行层

当前执行层不是主要瓶颈，大改收益低、风险高。

### 2. 继续主要靠 prompt 堆能力

prompt 还有价值，但单靠 prompt 很难突破感知、定位、验证的物理上限。

### 3. 一次性推翻六层重做

当前系统已经具备可用闭环和产品骨架，更适合渐进式替换。

## 分阶段验收建议

### P0 验收

- 建立一组固定桌面任务样本
- 统计点击命中率、任务成功率、验证误判率、恢复成功率
- 至少与当前主分支做一次量化对比

### P1 验收

- 核心逻辑测试稳定通过
- 历史漂移测试完成清理
- 模型切换行为一致性提高
- 高风险误操作显著减少

### P2 验收

- 复杂任务链路成功率提升
- 经验复用能够产生可量化收益
- 可以进行版本回归评估

## 最终建议

如果只选一个最关键方向，优先做：

> P0-1 感知增强 + P0-3 语义验证

原因是：

- 感知决定模型看到了什么
- 验证决定系统知不知道自己成功了

如果这两层不提升，定位和恢复再怎么补，也会反复受上游输入和下游判断能力限制。

如果只选一个最稳妥的工程推进方式，建议采用：

> 以 P0 为主线，穿插做 P1-2 测试体系重整

这样可以避免在核心能力升级过程中，测试系统继续和实现脱节。
